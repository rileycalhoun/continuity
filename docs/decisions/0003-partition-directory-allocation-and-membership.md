# ADR 0003: Partition directory, allocation, and membership changes

- **Status:** Accepted
- **Date:** 2026-07-11
- **Extends:** ADR 0001 and ADR 0002

## Context

Continuity worlds may expand indefinitely as new chunks generate, while operators must be able to add capacity or remove servers without changing the logical world. Deriving ownership directly from the current server list would make membership changes silently remap live world state and trigger uncontrolled migrations.

The system therefore needs stable partition identities, a durable ownership directory, safe first-use allocation, and explicit server lifecycle behavior.

## Decision

### Partition identity

Continuity divides each world and dimension into fixed rectangular partitions containing whole chunks.

A partition is identified independently of any server by its world, dimension, and grid coordinates. Its exact dimensions are configurable rather than hardcoded into the network protocol. Changing the dimensions of an initialized world requires an explicit future migration procedure.

Chunk coordinates map deterministically to a partition using mathematical floor division:

~~~text
partition_x = floorDiv(chunk_x, partition_width)
partition_z = floorDiv(chunk_z, partition_depth)
~~~

Mathematical floor division is required so negative chunk coordinates map correctly.

### Durable directory and live authority

The SQL database stores the authoritative partition directory. A materialized partition record contains enough information to identify and fence ownership, including:

~~~text
world_id
dimension
partition_x
partition_z
owner_server_id
ownership_epoch
state
storage_version
~~~

Redis stores server presence, cached directory data, short-lived transfer state, and the live ownership lease associated with the durable assignment and epoch.

Only the active coordinator may create assignments or change partition ownership. A Continuity Server must never claim authority merely because it loaded or received traffic for a chunk.

The partition key is unique in SQL. Concurrent first-access attempts therefore resolve to one directory record and one initial owner through an atomic transaction or conditional insert.

### Lazy materialization and chunk generation

Continuity does not preallocate partitions for the unbounded world. A logical partition remains virtual until something first approaches or requests data from it.

On first use:

1. The proxy calculates the partition key.
2. The coordinator atomically creates or reads its directory record.
3. The scheduler selects an eligible Continuity Server.
4. The selected server prepares the partition.
5. The proxy permits entry after the partition is ready.

An ungenerated chunk inside an existing partition is generated by that partition's current owner. Chunks are not assigned independently from their containing partition, and only the authoritative owner may generate or persist them.

Initial allocation should prefer a healthy server already owning adjacent partitions when capacity permits, reducing cross-server boundaries. Otherwise it selects the best healthy server using a load and capacity score.

### Adding servers

A new server progresses from JOINING to ACTIVE after connecting, authenticating, verifying protocol compatibility, and reporting capacity.

Becoming active does not recalculate existing assignments. Existing partitions remain sticky. The server becomes eligible for new partitions and may receive existing partitions through explicit, gradual rebalancing.

Rebalancing considers at least:

- Milliseconds per tick
- Active chunks and entities
- Player count
- Relative server capacity
- Estimated partition load
- Spatial adjacency
- The configured migration budget

Player count alone is not a sufficient load signal. The number of concurrent migrations is bounded and migrations are rate-limited.

### Removing servers

Graceful removal follows:

~~~text
ACTIVE -> DRAINING -> OFFLINE
~~~

A draining server receives no new assignments, remains authoritative for its current partitions, and migrates them away. It may become offline only after it owns no partitions and has no transfer in progress.

Unexpected failure follows:

~~~text
ACTIVE -> SUSPECT -> FAILED
~~~

The coordinator does not grant replacement authority until the failed owner's live lease has expired. It then commits a new owner with a higher ownership epoch. All consumers of authoritative mutations must reject work carrying an older epoch.

### Partition migration

Moving an existing partition is an explicit state transition:

1. The destination prepares the partition and required world data.
2. The source remains authoritative while preparation occurs.
3. The source freezes mutations at a tick boundary and transfers its final changes.
4. The coordinator conditionally commits the new SQL owner and increments the epoch.
5. The destination receives the live lease for the new epoch.
6. The source unloads the partition and can retain only non-authoritative data.

Membership changes never move a partition implicitly. A future automatic rebalancer must use this same migration operation rather than creating a separate ownership path.

### Failure durability

Graceful migration can preserve current state, but recovery after an unexpected process or machine failure is limited by the last durable or replicated state.

This ADR does not choose the physical partition-storage mechanism, journaling strategy, replication strategy, recovery point objective, or recovery time objective. Those require a separate decision. Continuity must not claim zero-data-loss failover unless the selected persistence design actually provides it.

[ADR 0004](0004-owner-local-storage-and-boundary-visibility.md) subsequently selects owner-local writable files with separate durability replication, while leaving the exact replication target, journal format, synchronization mode, and recovery objectives undecided.

## Consequences

### Benefits

- Adding or removing servers does not unexpectedly remap the entire world.
- New world areas are allocated without precreating an infinite partition map.
- Durable assignments survive Redis loss and process replacement.
- Live leases and ownership epochs fence stale servers.
- Manual and future automatic rebalancing share one safe migration primitive.
- Spatial locality can reduce cross-server boundary traffic.

### Costs and risks

- The coordinator becomes responsible for serialized assignment changes.
- Partition migration requires data transfer, a brief mutation freeze, and recovery logic.
- Load-aware placement requires trustworthy metrics beyond player count.
- Exact storage and crash-recovery guarantees remain unresolved.
- Changing partition dimensions after world initialization requires a dedicated migration.

## Rejected alternatives

- **Recalculate ownership from the current server list:** makes routine membership changes implicitly migrate live state.
- **Pure consistent hashing for existing partitions:** limits remapping but still changes ownership automatically and ignores migration readiness and spatial locality.
- **Allocate every chunk independently:** creates excessive directory entries and increases boundary and migration complexity.
- **Preallocate the world:** is unnecessary and cannot cover an effectively unbounded coordinate space.
- **Allow servers to claim unowned chunks themselves:** permits races and conflicting authority.
- **Stop a server before draining it:** risks unavailable partitions and loss of non-durable state.

## Compliance

An implementation conforms to this decision only if:

- Partition identities are independent of server identities.
- Membership changes do not implicitly change ownership.
- Every ownership change is serialized by the coordinator and increments the fencing epoch.
- SQL retains the durable directory while Redis represents live leases and cached state.
- Only the recorded live owner can generate or persist chunks.
- New partitions are materialized atomically.
- Graceful removal drains all partitions before shutdown.
- Migration and failure tests cover duplicate requests, concurrent first access, expired leases, stale epochs, coordinator restarts, and source or destination failure at every transition.
